## 構造の特徴

### 1. 標準的な構成パターン

**導入 → 課題 → 解決 → まとめ の流れ**

すべての記事がこの構造に従っている:

1. **導入(背景説明)**: 自分自身の経験や現実の課題から始まる
2. **問題提起**: 解決すべき技術的課題を明確化
3. **解決策の詳細**: 実装コード例付きで具体的に説明
4. **まとめ**: 学習ポイントをリストアップ

**例: 2026-01-12_my-playwright-was-wrong.md**
- 導入: 生成AIとの開発、過去コード流用の合理性
- 問題: `page.evaluate()` は動くが「正しく動く」のか？
- 解決: `locator` の設計思想と Actionability Checks
- まとめ: 学んだこと6点のリスト

```markdown

- 過去のコードを流用することは効率的で合理的な判断だが、「動く」と「正しく動く」の間には設計思想の違いがある
- `page.evaluate()` は動作するが、要素の操作可能性を確認しないため不安定なテストにつながる可能性がある
- `locator` は要素がクリック可能になるまで自動で待機し、Actionability Checks により安定性を向上させる

- 公式ドキュメントには「なぜこの方法が推奨されるのか」という設計思想が書かれており、単なるAPIリファレンス以上の価値がある
- 日常的に使うツールについては、一度は腰を据えてドキュメントを読む価値がある
```

**段階的な説明**

```

```

**用語の丁寧な説明**

```
`page.evaluate()` は、ブラウザコンテキスト[^ブラウザコンテキスト]内でJavaScriptコードを実行するためのメソッドです。
指定された関数は、ページ上で直接実行され、DOM要素にアクセスして操作を行います。

[^ブラウザコンテキスト]: ブラウザの実行環境のこと。ページ上でJavaScriptを直接実行する際の文脈を指します。
(2026-01-12_my-playwright-was-wrong.md より)
```


```
これには、要素が表示されているか、クリック可能な状態であるかなどのチェックが含まれます。
```

## 思想・価値観の特徴

### 1. AI文章への明確な嫌悪感

**「AIを24時間365日使う人間が、AI生成文章に最も嫌悪感を抱いている」という逆説**

使い手だからこそ分かる「手抜きの透視能力」を持っている。

**具体的なトリガー:**
- Markdown記法の残留(`**強調**`や`---`がそのまま表示)
- 不自然な文末記号(日本語の文末に「:」コロンが使用)
- 機械的な誘導(「詳しくはこちら👉」などの定型句)
- 型定義がない(TypeScriptファイルなのに型定義がない等、明らかに動かないミス)

**嫌悪感の3層構造:**

```
Layer 1: 不気味の谷(Uncanny Valley)
  ↓ 「なんか臭うな」という直感的違和感

Layer 2: 共感性羞恥(Vicarious Embarrassment)
  ↓ 「推敲もせずに出すのかよ」という手抜きへの怒り

Layer 3: 情報汚染への危機感(Information Pollution)
  ↓ 「間違った情報を垂れ流しにする」ことへの憤り
```

**感情表現:**

```

```


- 「私」を主人公にした一人称視点
- 読者への問いかけと共感の形式

### 構造

- 箇条書きによる学習ポイントのまとめ

### 思想・価値観
- AI文章への明確な嫌悪感(推敲しないことへの怒り)

これらの特徴により、技術情報を正確に伝えながらも、読者と共感し、親しみやすさを保ち、かつ明確な価値観を持った独特のスタイルを形成している。
