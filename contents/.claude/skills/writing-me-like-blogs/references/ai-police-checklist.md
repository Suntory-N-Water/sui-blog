## 思想的背景

### AI文章への明確な嫌悪感

「AIを24時間365日使う人間が、AI生成文章に最も嫌悪感を抱いている」という逆説。

使い手だからこそ分かる「手抜きの透視能力」を持っている。

### 嫌悪感の3層構造

```
Layer 1: 不気味の谷(Uncanny Valley)
  ↓ 「なんか臭うな」という直感的違和感

Layer 2: 共感性羞恥(Vicarious Embarrassment)
  ↓ 「推敲もせずに出すのかよ」という手抜きへの怒り

Layer 3: 情報汚染への危機感(Information Pollution)
  ↓ 「間違った情報を垂れ流しにする」ことへの憤り
```

### 基本姿勢

```
AI臭い文章を見た瞬間に萎える。「またAIね」という感覚。
```

```
一番ムカつくのは、明らかに生成AIで作っているのに「これは私の文章です」という顔をしているケース。
```

```
推敲もできないなら載せるなよ。
```

## 具体的なトリガー

以下の特徴が1つでも見られる場合、AI生成文章の疑いが強い。すべて排除すること。

### Markdown記法の残留

❌ **NG例**:

```
これは**非常に重要**な点です。
詳細は---で区切られています。
```

✅ **OK例**:

```
これは非常に重要な点です。
詳細は以下の通りです。
```

**対策**: マークダウン記法が文章中にそのまま残留していないか確認する。特に `**強調**` や `---` がそのまま表示されていないか検証する。

### 不自然な文末記号

❌ **NG例**:

```
次の手順に従ってください:
以下の項目を確認してください:
```

✅ **OK例**:

```
次の手順に従ってください。
以下の項目を確認してください。
```

**対策**: 日本語文末に「:」(コロン)を使用しない。箇条書きの導入部には「。」を使用する。

### 機械的な誘導

❌ **NG例**:

```
詳しくはこちら👉
さらに詳しく知りたい方はこちらをクリック!
```

✅ **OK例**:

```
詳しくは以下のリンクを参照してください。
```

**対策**: 「詳しくはこちら👉」のような定型句を使用しない。絵文字による誘導も避ける。

### 型定義がない

❌ **NG例**:

```typescript
// TypeScriptなのに型定義がない
function processData(data) {
  return data.map(item => item.value);
}
```

✅ **OK例**:

```typescript
function processData(data: Array<{value: string}>): string[] {
  return data.map(item => item.value);
}
```

**対策**: コード例では適切な型定義を含める。特にTypeScriptのコード例では型定義の省略は明らかなミスとして認識される。

## 逆説的な発見：不完全性の価値

### タイポ・誤字脱字への親近感

```
AIは誤字脱字を絶対しない(確率で出しているから)。
タイポや誤字脱字を見ると「まだこっちの方が人間味があっていいよね」と感じる。
```

過去、誤字脱字は「恥ずべきミス」だったが、AI時代では誤字脱字が「人間の証明」になる。

**注意**: これは誤字脱字を推奨するものではない。推敲は必須である。ただし、完璧すぎる文章は逆に不自然さを生む可能性がある。

## エビデンス重視への転換

### ここ1年で持論をガンガン出すように変化

```
公式ドキュメント、一次情報、実際のコードの中身と意図を必ず書く。
```

AI文章は「それっぽいこと」を確率的に出力するため、人間は「本当にそうなのか?」を検証する役割にシフトする。

### 実践例

```
公式ドキュメントには「なぜこの方法が推奨されるのか」という背景が書かれています。
単なるAPIリファレンスではなく、設計思想やベストプラクティスが凝縮されています。
```

```
https://playwright.dev/docs/evaluating

https://playwright.dev/docs/locators

https://playwright.dev/docs/actionability
```

**対策**: 技術的な主張には必ず公式ドキュメントやソースコードへのリンクを含める。

## 「人間臭いもの」「尖り散らかしたこと」への渇望

### AI文章にはできない「尖り」を求める

```
個人の技術記事には思想が載っていると人間味がある。
世相が載っている記事を見たい。
```

```
AI文章は「尖り散らかしたこと」はできない。
```

### 実践例

```
やはり俺のPlaywrightはまちがっている。
```

「やはり俺の青春ラブコメはまちがっている。」をパロディにした尖ったタイトル。

```
しかし、偶然にも公式ドキュメントを最初から読み直す機会を得ました。
Claude Codeにベストプラクティスを読み込ませるためのAgents Skillsを作成する必要があったからです。
```

具体的な体験と動機を明示する。

**対策**: 個人的な経験、具体的な動機、明確な価値観を文章に込める。

## AI活用に対する姿勢

### AIは積極的に活用するが、推敲は必須

```
「文章は人間が書くべき」: 100%は思わない。良いものはどんどん活用したい。
AIの補助で思考を素早く出せるのは問題ない。
```

```
「推敲は当然すべき」: 間違っていることや伝わりづらい表現があれば変えた方がいい。
```

```
頭で思っていることがそのまま出るなら「手抜き」ではない。
気持ち悪い点は、頭で思っていることと実際にアウトプットされる言葉が一致していないこと。
```

### 実践例

```
図解作成用の専用プロンプトを使用し、上記のフロントマターにある yaml を生成します。
このプロンプトは Gemini の Gem に定義しており、記事の Markdown ファイルを入力するだけで自動生成できます。
```

AIを活用した図解生成を明示的に説明している。

**対策**: AI活用は明示し、推敲を徹底する。

## 価値観の明確な表明

### 「こうあるべき」という論調を持つ

```
極端かもしれないが、「こうあるべき」という論調になってしまう。
```

```
「潔癖症だ」という批判: 全然潔癖症で構わない。あくまで自分の思想。
「こういった価値観はどうですか?」と伝えたいだけ。
```

### 実践例

```
「動けばよい」という判断は間違っていません。
しかし「正しく動く」ことを知った今、少し時間をかけてでも、公式ドキュメントを上から下まで読む価値があると私は考えます。
```

明確に自分の価値観を表明している。

**対策**: 技術的な判断に対する自分の価値観を明確に述べる。

## 最終チェックリスト

記事を公開する前に、以下の項目をすべて確認する。

- [ ] Markdown記法が文章中にそのまま残留していないか
- [ ] 日本語文末に「:」(コロン)を使用していないか
- [ ] 「詳しくはこちら👉」のような機械的な誘導を使用していないか
- [ ] コード例に適切な型定義が含まれているか(TypeScriptの場合)
- [ ] 技術的な主張に公式ドキュメントやソースコードへのリンクが含まれているか
- [ ] 個人的な経験や具体的な動機が記述されているか
- [ ] 自分の価値観が明確に表明されているか
- [ ] 「〜することができます」が連続使用されていないか
- [ ] 過度に丁寧で無機質な表現になっていないか
- [ ] 文章が自分の思考と一致しているか(推敲済みか)
