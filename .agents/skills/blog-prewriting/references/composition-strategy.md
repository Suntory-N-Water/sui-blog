## 論調の使い分け

技術記事は、対象読者や内容に応じて 2 つの論調を使い分ける。後述の「共感と裏切り」「SCQ + Deep Empathy」は**硬めの論調**にのみ適用し、ふわっとした論調には適用しない。

- **硬めの論調**: 初めてその技術に触れる読者でも理解できるよう、前提知識から丁寧に解説する。背景や歴史的経緯を含め、知識の基盤を固めることを重視する
- **ふわっとした論調**: 特定のアップデート情報や、簡単な「試してみた」系の記事など、前提知識の解説が冗長になるときは、要点を絞った簡潔な構成を目指す

## 記事パターンの分類

執筆する記事を以下の 3 パターンに分類し、「硬めの論調」か「ふわっとした論調」かを判断する。

### 1. 新仕様の解説記事(硬めの論調)

- **内容**: 既存の技術に追加された新しい仕様(例: Claude Code の Skill など)
- **コンテキスト**: **必須(共感と裏切りを適用)**。「既存技術とは何か」「なぜこの仕様が追加されたのか」といった背景説明に加え、現状のやり方を肯定したうえで新機能を提案する

### 2. 既存技術の活用記事(硬めの論調)

- **内容**: 複数の技術を組み合わせた応用的な内容
- **具体例**:
  - [GAS を使うと社内だけで使える Web サイトが簡単に作れるの知ってました? - Zenn](https://zenn.dev/sui_water/articles/502a6ef30b79e1)
  - [Anthropic 公式の MCP によるコード実行を serena MCP で試してみたけど無理だった件 - Zenn](https://zenn.dev/sui_water/articles/717eaa7e5b2425)
- **コンテキスト**: **必須(共感と裏切りを適用)**。「なぜこの技術を選んだのか」「どのような課題を解決するのか」といった基盤の共有がなければ、読者は記事についていけない

### 3. Tips・調査記事(ふわっとした論調)

- **内容**: 少し気を楽にした解説や、特定のエラー解決策、小規模な実践報告
- **具体例**:
  - [Notion の破壊的変更の記事](https://zenn.dev/sui_water/articles/717eaa7e5b2425)
  - [PreToolUse が failed with non-blocking status code 127 になったときにやったこと - Zenn](https://zenn.dev/sui_water/articles/ce32e569d738ea)
- **コンテキスト**: **任意(共感と裏切りは不要)**。記事の目的が特定の情報提供に絞られている場合、「Notion API とは?」のような冗長な説明はかえって読者の負担になる

## コンテキストファーストの原則

我々が執筆する記事は、単なる情報の羅列であってはならない。読者の理解を最大化するため、**コンテキストファースト**の原則を最も重要視する。

### なぜコンテキストが重要か

人間は、大量の情報を処理する際に**ストーリー形成を通じて理解・認知する**。記事の導入部で読者と執筆者の間の前提知識(コンテキスト)を揃えることで、記事全体の理解度が飛躍的に高まる。

この考え方は、Stack Overflow が常に問題の背景を説明してから解決策を提示する構成になっていることからも、その有効性がわかる。

### コンテキストを揃えるための導入部

導入部の役割は、**Situation(状況) → Complication(複雑化・課題) → Question(疑問・本題)** というストーリーを提示し、読者の関心を引きつけ、前提知識を揃えることにある。

- **コンテキストの具体例**:
  - 「このエラーは、〇〇というライブラリをバージョンアップした際に発生しました」(状況)
  - 「この技術は、従来の〇〇が抱えていたパフォーマンス問題を解決するために生まれました」(背景)

### 記事の基本構成テンプレート

コンテキストファーストの思想に基づき、以下を基本テンプレートとする。

1. **導入部**: コンテキストを揃える(硬めの論調なら SCQ + Deep Empathy)
2. **課題提起**: 記事が解決しようとしている具体的な問題
3. **解決策**: 課題を解決するための技術、ツール、手法の紹介
4. **実装例**: 解決策を具体的に示すコードや手順
5. **結論**: 記事全体の要点をまとめ、学びやクイズを提示
6. **参考文献**: 記事の信頼性を担保するための一次情報や公式ドキュメント

## SCQ + Deep Empathy 型(硬めの論調向け)

硬めの論調における導入部は、従来の SCQ に「深い共感」を加えた以下の流れを基本とする。

1. **Situation(状況)**
   - 読者が置かれている状況や、一般的な課題を描写する
   - 例: 「テストコードを書くことは品質担保のために重要だと言われています」
2. **Deep Empathy(現状維持の肯定)**
   - **重要**: ここですぐに解決策を提示しない。読者がその正論を実行できていない理由を代弁し、**「今のままでも仕方ない、それが合理的だ」と一度完全に肯定**する
   - 例: 「とはいえ、納期に追われる中でテストを書くのは困難です。仕様変更でテストが壊れる徒労感を考えると、『手動で確認した方が早い』と思うのは当然です。私もそう思います」
3. **Complication(部分的な裏切り)**
   - 完全に共感した基盤の上で、常識の枠組みの中で**「一部だけ」**新しい視点を提示する
   - 例: 「確かに全てをテストするのは非効率です。しかし、実は**『ログイン周り』の 1 箇所だけ**を自動化するだけで、リリースの心理的負担が激減することをご存知ですか?」
4. **Question(解決策への導入)**
   - その裏切りを実現するための具体的な手段を提示する

## 共感と裏切り

読者の行動を変える、あるいは深い納得感を生む記事を書くための中心概念。主に「硬めの論調」の記事で、導入部の SCQ 構成を強化するために使用する。

### 定義と意図

人間は「自分はそれなりにできている(自尊心)」と「本当はもっとできるのではないか(不安)」という矛盾する感情を抱えている。いきなり正論(新しい技術や手法)を突きつけると、読者は自尊心を守るために反発する。そのため、以下の手順で信頼を築き、関心を引きつける。

1. **全体の共感(肯定)**: 読者の現状や、新しい技術を取り入れていない理由(コスト、面倒さなど)を完全に肯定し、信頼の基盤を築く
2. **部分の裏切り(提案)**: 信頼を得たうえで、「実は一部だけを変えることで劇的に良くなる」という新しい視点を提示する

## アンチパターン

本手法において、以下の 3 つのアプローチは読者の「心の扉」を閉ざす原因となるため避ける。

### 1. いきなり危機感を煽る(現状の否定)

- **内容**: 導入部で「あなたは〇〇を使いこなせていますか?」のように問いかけ、「今のままではダメだ」と指摘すること
- **理由**: 読者は「自分はそれなりにできている」という自尊心を持っている。冒頭から否定されるとプライドが傷つき、反発心が生まれる

### 2. 相手を「無知」な前提で扱う

- **内容**: 「〇〇を知らないだろう」という前提で、誰もが知っているような基礎知識を得意げに解説すること
- **理由**: 読者はすでに多くの情報を知っており、基礎的なハウツーには飽きている。「そんなことは知っている」と思われ、信頼関係が崩れる

### 3. 解決策を急ぐ(共感の省略)

- **内容**: 現状の苦労や背景への共感(Deep Empathy)を飛ばし、すぐに便利なツールや解決策を提示すること
- **理由**: 「現状維持が最適解だ」と思っている読者の感情に寄り添う前に解決策を出しても、単なる押し売りに見える。「自分と同じ感情を共有している仲間だ」という認識を作るプロセスが不可欠
